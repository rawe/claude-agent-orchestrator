## What Is Still Open

### Next Task: Unified Stop Action with Smart Shutdown

#### Status Semantics (Canonical Definitions)

These are the **authoritative** definitions. No ambiguity.

| Status | Meaning | Who decides? | Process? |
|--------|---------|-------------|----------|
| `pending` | Session created, no executor bound yet | System | Not started |
| `running` | Agent is actively processing a turn | System | Alive, busy |
| `idle` | Agent completed a turn, process alive, waiting for next input | Agent | Alive, waiting |
| `stopping` | Shutdown in progress (graceful or force) | System | Alive, terminating |
| `finished` | **Agent exited cleanly on its own terms.** Either the agent decided it's done, or the agent received a graceful shutdown request and exited cleanly (exit 0). | Agent | Exited |
| `stopped` | **System force-terminated the process.** The agent was interrupted — it did not get to finish on its own terms (SIGTERM/SIGKILL). | System | Exited |
| `failed` | Agent process crashed (non-zero exit). | Agent (crash) | Exited |

**Key distinction**: `finished` = the agent had the final say (clean exit). `stopped` = the system had the final say (force kill).

#### The Problem

The dashboard currently has two actions: **Stop** (force-kill) and **Delete** (remove from DB).
There is no way to gracefully end an idle session. The runner's `_handle_stop()` always
sends SIGTERM/SIGKILL, even for idle sessions where the NDJSON shutdown protocol could be
used for a clean exit.

#### The Solution: One Smart "Stop" Action

No separate "Finish" button needed. A single **Stop** action that behaves differently
based on session state:

| Session State | What "Stop" Does | Method | End Status |
|--------------|-----------------|--------|------------|
| `running` | Force-kill (agent is busy, can't send NDJSON) | SIGTERM → SIGKILL | `stopped` |
| `idle` | Graceful shutdown (agent is waiting, can receive NDJSON) | `{"type":"shutdown"}` via NDJSON stdin → wait for clean exit → timeout: escalate to SIGTERM | `finished` (if clean exit) or `stopped` (if timeout/force) |

This means:
- **One user-facing action** ("Stop") with smart behavior
- **One API endpoint** (`POST /sessions/{id}/stop` — already exists)
- End status **reflects reality**: `finished` if agent exited cleanly, `stopped` if force-killed

#### Implementation Across the Stack

**1. Coordinator (`servers/agent-coordinator/main.py`)**

No API changes needed. `POST /sessions/{id}/stop` already works for both `running` and
`idle`. It sets status to `stopping` and queues a stop command. The coordinator does NOT
need to know about graceful vs force — that's a runner concern.

However, the stop command queue may need a `mode` field so the runner knows the intent:
```
stop_command_queue.add_stop(runner_id, session_id, mode="graceful"|"force")
```
Or the runner can infer mode from session state (idle → graceful, running → force).
**This is a design decision for the planner agent.**

**2. Runner — Stop Command Queue Transport (`servers/agent-runner/lib/poller.py`)**

Current `_handle_stop()` always does SIGTERM → SIGKILL. Needs to be updated:

```
def _handle_stop(self, session_id):
    entry = self.registry.get_session(session_id)
    if entry is None:
        return

    run_id = entry.current_run_id

    if run_id is None:
        # Session is idle — graceful shutdown via NDJSON
        entry.executor.send_shutdown()        # {"type":"shutdown"}
        # Wait for clean exit (timeout e.g. 10s)
        try:
            entry.process.wait(timeout=10)
            # Clean exit → report "finished"
            self.api_client.report_session_status(runner_id, session_id, "finished")
        except TimeoutError:
            # Didn't exit cleanly → escalate to force
            entry.process.terminate()  # SIGTERM
            entry.process.wait(timeout=5)
            self.api_client.report_session_status(runner_id, session_id, "stopped")
    else:
        # Session is running — force kill (existing behavior)
        entry.process.terminate()
        ...
        self.api_client.report_stopped(runner_id, run_id, signal="SIGTERM")
```

**Key**: `send_shutdown()` already exists in `executor.py`. The NDJSON shutdown protocol
is already implemented in the executor. The runner just needs to USE it for idle sessions
instead of always force-killing.

**3. Runner — Supervisor (`servers/agent-runner/lib/supervisor.py`)**

The supervisor's `_handle_persistent_exit()` already handles idle process exit and
reports `finished` (exit 0) or `failed` (non-zero). After the poller sends shutdown
and the process exits, the supervisor's existing exit handling should work.

**Potential race**: If the poller sends shutdown AND the supervisor detects exit, both
might try to report status. Needs deduplication. **Design decision for planner agent.**

**4. Dashboard (`apps/dashboard/src/`)**

Single "Stop" button (already exists on SessionCard for `running` and `idle`). No new
buttons needed. The UI just needs to:
- Show appropriate confirmation message based on status ("Stop this running session?"
  vs "End this idle session?")
- React to status changes via SSE (stopping → finished or stopped)

Additional fixes needed:
- `AgentSessions.tsx:144`: `isRunning` check should include `idle` for timeline indicator
- `Chat.tsx`: Verify SSE status updates correctly reflect transitions
- Delete action: Should warn or prevent deleting sessions with live processes

**5. Chat UI (`apps/chat-ui/src/`)**

- When session transitions to `finished`, disable input and show "Session ended" or
  offer "New Chat"
- The `AgentStatus` type already has `'finished'` — just ensure SSE events map correctly

**6. Delete Safety (`DELETE /sessions/{id}`)**

Currently deletes from DB without stopping any live process. Needs to be safe:
- If session status is `running` or `idle` → reject with 409 ("Stop session first")
  OR auto-stop then delete
- If session status is terminal (`finished`/`stopped`/`failed`) → delete normally
- **Design decision for planner agent.**

## Session Status State Machine (Reference)

```
pending → running → idle → running → idle → ... → finished  (agent decides it's done)
                  ↘ failed  ↗       ↘ failed
                  ↘ stopping → stopped  (force-killed)
         idle → stopping → finished  (graceful shutdown, clean exit)
                         → stopped   (graceful timed out, force-killed)
```

7 statuses: `pending`, `running`, `idle`, `stopping`, `finished`, `stopped`, `failed`

## How to Start Components (Dev Mode)

All scripts are in `scripts/`. Each runs a component in the foreground. The team lead
should start them as **background tasks** and monitor their output for errors.

```bash
# 1. Agent Coordinator (must start first — others connect to it)
#    - Runs on http://localhost:8765
#    - API docs at http://localhost:8765/docs
./scripts/start-coordinator.sh

# 2. Dashboard (Vite dev server with hot reload)
#    - Runs on http://localhost:3000
#    - .tsx/.ts changes take effect immediately (no restart)
./scripts/start-dashboard.sh

# 3. Agent Runner with Claude SDK Executor
#    - Uses profile: claude-sdk-executor (lifecycle: persistent)
#    - Always use this, NOT start-runner-claude-code.sh (legacy)
./scripts/start-runner-claude-sdk.sh
```

**Restart rules:**
- **Coordinator** (Python): must restart after code changes
- **Dashboard** (TypeScript/Vite): hot reloads automatically, no restart needed
- **Runner** (Python): must restart after code changes. Runner ID is deterministic
  (same hostname + project_dir + executor_profile = same ID), so restarting may
  fail if the coordinator still has the old registration. Restart coordinator first
  if this happens.
