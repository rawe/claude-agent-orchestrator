## What Was Found

### Key Architectural Decisions

1. **run_id vs session_id at each boundary**: Runner→Coordinator uses run_id. Executor→Coordinator uses session_id. Runner registry keyed by session_id. The executor never needs run_id.

2. **Stop = abort session permanently**: Cannot resume after stop (SDK limitation). Stop interface changes from run_id to session_id. External clients know session_id, may not know current run_id.

3. **Session status `idle`**: Between turns, the session is `idle` (process alive, resumable). `finished` means permanently done. One-shot executors never use `idle`.

4. **No run_id in NDJSON protocol**: The executor is session-pure. The runner maps `turn_complete` to the current `run_id` via the registry. The turn protocol is strictly sequential — no correlation ID needed.

5. **Bind call stays**: It delivers hostname + executor_profile + running status. executor_session_id is stored but irrelevant for multi-turn resume.

### Consistency Issues Found and Resolved

- STOP-COMMAND-REDESIGN.md uses old method names (`get_session_process`) — use REGISTRY-REDESIGN.md names (`get_session`)
- PHASE2-RUNNER-ARCHITECTURE.md stdout reader reads `msg["run_id"]` — superseded, use `entry.current_run_id`
- `send_turn()` in PHASE2-RUNNER-ARCHITECTURE.md includes run_id — implement WITHOUT run_id per NDJSON-PROTOCOL-SIMPLIFICATION.md

### Review Findings Addressed

- Phase A must include ALL registry callers (poller, supervisor, agent-runner script)
- Phase C (NDJSON simplification) merged into Phase B
- Resume-while-busy guard: reject if `current_run_id is not None`
- `_reported_runs` dedup set needs explicit lock
