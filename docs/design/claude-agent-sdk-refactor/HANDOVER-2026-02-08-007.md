## How to Start Components (Dev Mode)

All scripts are in `scripts/`. Each runs a component in the foreground. The team lead
should start them as **background tasks** and monitor their output for errors.

```bash
# 1. Agent Coordinator (must start first â€” others connect to it)
#    - Runs on http://localhost:8765
#    - API docs at http://localhost:8765/docs
#    - Auth disabled, CORS open
./scripts/start-coordinator.sh

# 2. Dashboard (Vite dev server with hot reload)
#    - Runs on http://localhost:3000
#    - Changes to .tsx files take effect immediately (no restart needed)
#    - Auth disabled
./scripts/start-dashboard.sh

# 3. Agent Runner with Claude SDK Executor (the refactored one)
#    - Uses profile: claude-sdk-executor (lifecycle: persistent)
#    - Connects to coordinator at :8765
#    - This is the executor we refactored â€” use this one, NOT start-runner-claude-code.sh
./scripts/start-runner-claude-sdk.sh
```

**Important**: `start-runner-claude-sdk.sh` runs the refactored executor (`--profile claude-sdk-executor`).
`start-runner-claude-code.sh` runs the OLD legacy executor. Always use the SDK one.

**Dashboard hot reload**: Since the dashboard uses Vite, any `.tsx`/`.ts` file change is picked
up automatically. The user can see changes in the browser without restarting. This makes it
ideal for the team lead to start the dashboard in background and let the dev agent make
changes that the user can verify live.

**Coordinator restart**: If coordinator code is changed (Python), it must be restarted.
The team lead should stop and restart the background task. The runner will reconnect
automatically (it retries on connection failure).
